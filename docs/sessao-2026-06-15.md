## Visão geral

A loja perdia muitas vendas por abandono de carrinho: o usuário precisava criar conta com e-mail e senha antes de comprar. A solução foi o sistema **Compra Rápida** — o usuário compra inserindo apenas nome e celular, sem criar conta.

O celular é a chave de vinculação: se o mesmo número for usado para criar uma conta depois, todos os pedidos feitos como guest aparecem automaticamente na nova conta.

Além disso, toda a interface do carrinho foi redesenhada, o modal de checkout ganhou uma versão mobile-first completa, e foi adicionado um botão flutuante de carrinho (pílula no rodapé) para melhorar a conversão.

## Compra Rápida — Guest Checkout

### Arquitetura

```
Usuário adiciona ao carrinho
    ↓
Clica "Comprar agora"
    ↓
GuestCheckoutModal — bifurcação:
  ├── "Já tenho conta" → login com redirect=/checkout
  └── "Compra Rápida" → formulário nome + celular
         ↓
  Appwrite: createAnonymousSession()
  localStorage.setItem('pedago_guest', { name, phone })
  auth.fetchMe() → detecta pedago_guest, mantém sessão
         ↓
  /checkout (CheckoutView)
  → lê guestData do localStorage
  → cria pedido com guestPhone
         ↓
  /checkout/success/:orderId
  → exibe banner "Crie conta com este celular"
  → onUnmounted: remove pedago_guest do localStorage
```

**Vinculação de conta futura:**
```
Usuário cria conta com o mesmo celular
    ↓
register-user (Appwrite Function)
    ↓
Query.equal('guestPhone', phone) → lista pedidos guest
    ↓
Para cada pedido: atualiza userId, limpa guestPhone,
  concede Permission.read(Role.user(userId)) no pedido,
  order_items e download_tokens
```

### Fluxo completo

1. **Carrinho cheio** → botão "Comprar agora" (guest) abre `GuestCheckoutModal`
2. **Tela 1 — seleção**: "Compra Rápida" (destacado) ou "Já tenho conta"
3. **Tela 2 — formulário**: nome + celular com máscara `(11) 99999-9999`
4. **Submit**: `createAnonymousSession()` → salva `pedago_guest` → `fetchMe()` → navega para `/checkout`
5. **Checkout**: lê `guestData`, exibe "Compra rápida · Nome", envia `guestPhone` na criação do pedido
6. **Success**: banner violeta convidando a criar conta; limpa `pedago_guest` ao sair da página

### Armadilhas críticas

#### 1. Ordem obrigatória: `pedago_guest` ANTES de `fetchMe()`

```ts
// ✅ CORRETO — fetchMe checa pedago_guest para decidir se mantém a sessão anônima
await account.createAnonymousSession();
localStorage.setItem('pedago_guest', JSON.stringify({ name, phone }));
await auth.fetchMe();

// ❌ ERRADO — fetchMe deleta a sessão anônima porque pedago_guest não existe ainda
await account.createAnonymousSession();
await auth.fetchMe();
localStorage.setItem('pedago_guest', JSON.stringify({ name, phone }));
```

#### 2. Três sistemas independentes de "phone required"

O projeto tem **três** verificações de telefone obrigatório, todas precisam excluir guests:

```ts
// 1. router/index.ts — guard de rota
const isGuest = !!localStorage.getItem('pedago_guest');
if (!isGuest && auth.isLoggedIn && !auth.user?.phone && ...) { ... }

// 2. PublicLayout.vue — modal local
const showPhoneModal = computed(() =>
  auth.isLoggedIn && !auth.user?.phone && !phoneDismissed.value &&
  !AUTH_PATHS.some(p => route.path.startsWith(p)) &&
  !localStorage.getItem('pedago_guest')   // ← obrigatório
);

// 3. PhoneRequiredModal.vue — modal global via App.vue
const show = computed(() =>
  auth.isLoggedIn && !auth.user?.phone &&
  !AUTH_ROUTE_NAMES.includes(route.name as string) &&
  !localStorage.getItem('pedago_guest')   // ← obrigatório
);
```

#### 3. `pedago_guest` só é removido no `onUnmounted` do CheckoutSuccessView

Remover antes (ex: no CheckoutView após criar o pedido) quebra o fluxo porque a success page ainda precisa da chave para saber que é um guest e exibir o banner correto.

#### 4. Sessões anônimas persistem no cookie do browser

Se o usuário abandonar no meio do fluxo e voltar ao site, o `fetchMe()` encontra a sessão anônima. A lógica no store trata isso:

```ts
// auth.store.ts — em fetchMe()
if (!profile && !authUser.email) {
  if (localStorage.getItem('pedago_guest')) {
    // Guest legítimo: mantém sessão, define user mínimo
    user.value = { id: authUser.$id, name: '', email: '', role: 'CUSTOMER', ... };
  } else {
    // Sessão anônima órfã: limpa
    await account.deleteSession('current').catch(() => {});
    user.value = null;
  }
  return;
}
```

### Appwrite — atributo guestPhone

Criado via `scripts/add-guest-phone-attr.mjs`:

- **Collection**: `orders` (em `pedago-db`)
- **Atributo**: `guestPhone` — String, tamanho 20, opcional, não obrigatório
- **Índice**: `guestPhone_idx` — tipo Key (para `Query.equal`)

O script está em `scripts/add-guest-phone-attr.mjs` e pode ser re-executado com segurança (ignora erros 409 de atributo já existente).

## CartFab — Botão flutuante

**Arquivo**: `apps/web/src/components/catalog/CartFab.vue`

Pílula flutuante que aparece no rodapé da tela (centralizada) quando há itens no carrinho.

### Comportamento

- Aparece quando `cart.count > 0` AND `!cart.isOpen` AND rota não é checkout/success
- **Mobile**: `bottom: calc(68px + env(safe-area-inset-bottom, 0px))` — acima da bottom nav
- **Desktop**: `bottom: 24px` — rodapé centralizado
- Desaparece automaticamente quando o drawer é aberto
- Badge de contagem com animação bounce quando o número muda

### Por que "rail + botão" (não `position:fixed` com `transform`)

O padrão original usava `left: 50%; transform: translateX(-50%)` no próprio botão. Isso causava o bug de "às vezes não clica" — em alguns browsers/iOS, `transform` no elemento `position:fixed` desalinha a área de clique visual do elemento interativo.

**Solução correta:**

```html
<!-- Rail: full-width, pointer-events:none, centraliza com flex -->
<div class="fab-rail">
  <!-- Botão: pointer-events:auto, zero transform de posicionamento -->
  <button class="fab-btn" @click="cart.openCart()">...</button>
</div>
```

```css
.fab-rail {
  position: fixed;
  left: 0; right: 0;
  display: flex;
  justify-content: center;
  pointer-events: none; /* não bloqueia cliques no conteúdo abaixo */
}

.fab-btn {
  pointer-events: auto;
  touch-action: manipulation;          /* elimina delay 300ms iOS */
  -webkit-tap-highlight-color: transparent;
}
```

### Onde é montado

```vue
<!-- PublicLayout.vue -->
<CartDrawer v-if="cart.isOpen" @close="cart.closeCart()" />
<CartFab />
```

O `CartFab` usa `<Teleport to="body">` — renderizado diretamente no `<body>`, nunca dentro de um container com `overflow:hidden`.

## GuestCheckoutModal — redesign mobile-first

**Arquivo**: `apps/web/src/components/checkout/GuestCheckoutModal.vue`

### Problemas corrigidos na versão mobile

| Problema | Correção |
|---|---|
| `text-sm` (14px) nos inputs → zoom automático no iOS | `font-size: 16px` explícito nos inputs |
| `translateX(-50%)` no card → desalinhamento de clique | Removido; card usa `w-full max-w-sm` sem transform |
| Modal com cantos redondos em todos os lados no mobile | `border-radius: 24px 24px 0 0` no mobile (bottom sheet), redondo em todos no desktop (`sm:rounded-3xl`) |
| Trust row `text-[11px]` ilegível, overflow em 320px | Grid 3 colunas com `trust-pill`, `text-xs`, cada item no seu espaço |
| Step pills `text-[10px]` cortando | `text-xs` + separador `›` em vez de SVG — não quebra em telas estreitas |
| Formulário não resetava ao fechar o modal | `watch(modelValue)` reseta campos após fechar |
| Sem auto-focus ao entrar no formulário | `watch(screen)` + `nextTick()` + `nameInputRef.focus()` |
| Sem handle bar visual (bottom sheet) | Barra cinza no topo em mobile: `<div class="w-10 h-1 bg-gray-200 rounded-full" />` |
| Touch targets pequenos | CTA: `min-height: 56px`, inputs: `py-4` |
| Sem iOS safe area | `env(safe-area-inset-bottom)` ao fim de cada tela |

### Progress steps (tela 2)

```html
<!-- Mostra contexto: Carrinho → Dados → Pagar -->
<div class="flex items-center gap-2">
  <div class="progress-step done">1</div>
  <div class="flex-1 h-0.5 bg-white/40" />
  <div class="progress-step active">2</div>

  <div class="progress-step">3</div>
```

### Acessibilidade

- `role="dialog"` + `aria-modal="true"` + `aria-label` dinâmico por tela
- `aria-hidden="true"` em todos os ícones decorativos
- `role="alert"` nas mensagens de erro
- `aria-label` específico em cada botão interativo
- `aria-busy` no CTA durante loading
- `autocomplete="name"` e `autocomplete="tel"` nos campos
- `inputmode="numeric"` no campo de celular (teclado numérico no mobile)

## Bugs corrigidos

| # | Sintoma | Causa | Correção |
|---|---|---|---|
| 1 | Guest era redirecionado para login | `pedago_guest` salvo DEPOIS do `fetchMe()` | Salvar ANTES do `fetchMe()` |
| 2 | `fetchMe()` não era chamado no modal | `user.value` ficava null após `createAnonymousSession()` | Chamar `await auth.fetchMe()` no modal |
| 3 | Modal de WhatsApp aparecia do nada | Sessão anônima órfã de tentativa anterior | `fetchMe()` deleta sessão anônima se não há `pedago_guest` |
| 4 | Modal de phone no `PublicLayout` ignorava guests | Computed `showPhoneModal` sem check de guest | Adicionado `!localStorage.getItem('pedago_guest')` |
| 5 | "Complete seu cadastro" ainda aparecia | `PhoneRequiredModal` global (App.vue) sem check de guest | Adicionado check no `show` computed do modal |
| 6 | Carrinho ficava aberto após ir ao checkout | `GuestCheckoutModal` não emitia `navigate` para fechar o drawer | Adicionado `emit('navigate')` antes de `router.push` |
| 7 | CartFab às vezes não respondia ao clique | `transform: translateX(-50%)` no elemento `fixed` desalinha área de clique | Padrão "rail + botão" — sem transform no elemento interativo |
